# Task 상태값 (Status)

## 3-값 모델 (spec 2026-06-03)

`tasks.status` 는 **등록 · 완료 · 취소** 3개뿐. 단일 진실은 `src/lib/types.ts`의
`TASK_STATUSES = ['등록', '완료', '취소']`.

| 값 | 의미 | 종결 여부 |
|---|---|---|
| `등록` | 기본값. 처리해야 할 활성 task. | 활성 |
| `완료` | 끝낸 task. `completed_at` 세팅. | 종결 |
| `취소` | 하지 않기로 한 task. 완료와 동급 종결. | 종결 |

- 기본값: DB `default '등록'`, 코드 `normalizeStatus` fallback `'등록'`.
- 종결 판정은 `isTaskDone(status)` 단일 헬퍼 (`완료 || 취소`). 인박스 활성
  카운트·오늘 진행률·사이드바 배지(`/api/tasks/count`)·무게(`getTaskWeight`
  호출부)·line-through 시각 처리 전부 이 기준 하나를 공유.
- 취소는 완료와 **동일한 제외 규칙**을 따른다 (활성 인박스/오늘/카운트에서 빠짐).

## 위임은 status가 아니라 필드 (결정)

과거에는 `위임`이 status 값이었다. 단순화하면서 **위임을 status에서 제거**하고
`tasks.delegate_to` 필드로만 표현한다. 근거:

- 위임은 "종결"이 아니다 — 위임해도 본인 인박스 흐름상 후속(팔로업)이 남는다.
  status 종결 취급(완료와 동급)이 사용자가 느낀 "등록/진행 혼란"의 일부였다.
- 위임 대상은 이미 `delegate_to` 라는 **독립 필드**로 존재했고, 인박스 필터의
  "위임 대상" 항목도 status가 아니라 `delegate_to` 기준으로 동작한다 (변경 없음).
- task-card의 "위임" 퀵액션(컨텍스트 메뉴 / ⋯ 드롭다운)은 status를 `위임`으로
  바꾸기만 하던 것이라 **제거**. 위임 대상 입력은 상세 패널 · 인라인 에디터 ·
  task-form의 `delegate_to` 입력에서 그대로 한다.

## `/inbox` 보기 칩 용어 일치 (결정)

- 보기 칩: **등록 · 보류 · 완료 · 휴지통**. 기존 "진행 중" 라벨을 status명
  `등록`으로 맞춰, 칩 용어와 카드 status 뱃지가 같은 단어를 쓰게 했다
  (사용자가 느낀 "등록/진행 오류"의 직접 원인).
- 취소된 task는 별도 칩 없이 **완료 칩(`done`) 뷰에 함께** 표시 — 가장 단순한
  쪽. 카드 status 뱃지가 `취소`로 떠 완료와 시각적으로 구분되고, 종결이라
  line-through도 함께 적용된다.

## DB

- CHECK 제약: `tasks_status_check check (status in ('등록','완료','취소'))`.
- 마이그레이션 이력: `004`(진행중/대기중 → 등록 + 위임/취소 허용),
  `007`(위임 → 등록 이전 + 제약을 3값으로 축소, `delegate_to`는 보존).
