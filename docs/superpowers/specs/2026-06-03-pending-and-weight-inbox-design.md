# 보류함 + 우선순위 제거 + 무게 인박스 — 설계

**날짜**: 2026-06-03
**상태**: 사용자 승인 완료 (브레인스토밍 세션)
**범위**: 보류(pending) 기능 신설, priority 필드 전면 제거, 인박스 마감일 기반 시각 위계

---

## 배경 / 문제

1. task가 인박스에 쌓이기만 함. "지금 안 할 것"을 치워둘 곳이 없어 우선순위 낮은 일이 시야를 차지함.
2. 모든 task가 같은 모양의 한 줄이라 긴급한 것과 사소한 것이 구분 안 됨 (사용자 표현: "눈에 확 안 들어옴").
3. 우선순위 필드는 자동 유입(Slack/Notion) task에서 전부 '보통'으로 들어와 사실상 죽은 데이터. 사용자가 직접 제거 결정.

## 결정 사항 (Q&A 요약)

| 질문 | 결정 |
|---|---|
| pending 동작 방식 | 별도 보류함. 인박스에서 완전히 사라지고 수동 복귀 (스누즈·자동복귀 아님) |
| 잊힘 방지 장치 | 없음 — 순수 수납함 (뱃지·리마인드 ❌) |
| sub-task 있는 task 보류 | 통째로 이동, 통째로 복귀 |
| ISSUE 보류 | 지원. 하위 task 전부와 함께 이동 |
| 보류된 ISSUE에서 개별 task 꺼내기 | **불가** — ISSUE 통째로만 복귀 (단순성 우선) |
| 무게 기준 | 마감일 단독. priority 필드는 제거 |
| 마감 없는 급한 일 | 마감일을 오늘로 직접 설정하는 것으로 갈음 |
| 시각 방향 | A안 "무게 리스트" (트리 구조 유지 + 줄별 시각 무게 차등) + B안에서 ISSUE 헤더 진행률만 차용 |

---

## 1. 보류함 (Pending)

### 데이터 모델

- `tasks.pending_at timestamptz null`, `issues.pending_at timestamptz null` 컬럼 추가 (마이그레이션 005)
- `pending_at != null` = 보류 상태. 휴지통(`is_deleted`)과 동일한 soft-flag 패턴, 두 플래그는 직교 — `is_deleted`가 우선 (삭제된 건 보류함에도 안 보임)
- status(등록/완료/위임/취소)와도 직교 — 보류 중에도 status 변경 가능 (예: Notion sync가 보류된 task를 완료 처리해도 보류 유지)

### 동작

- **task 보류**: 해당 task + 모든 하위 sub-task에 `pending_at` 설정. 인박스·/today·활성 카운트에서 즉시 제외
- **task 복귀**: task + 하위 전부 `pending_at` 해제. 원래 ISSUE 소속·계층·position 그대로 복원 (position은 건드리지 않음)
- **ISSUE 보류**: issue + 소속 task 전체(+sub-task)에 `pending_at` 설정. 인박스 트리에서 ISSUE 줄 자체가 사라짐
- **ISSUE 복귀**: issue 단위로만. issue + 소속 task 전체 해제
  - 단순화 수용: ISSUE 보류 이전에 개별 보류돼 있던 task도 ISSUE 복귀 시 같이 복귀됨 (보류 출처를 구분하지 않음)

### UI

- **사이드바**: "보류함" 메뉴 추가 (휴지통과 같은 급). 뱃지 없음
- **보류 진입점**: task 카드 hover 메뉴·상세 패널에 "보류" 액션, ISSUE 줄 ⋯ 메뉴에 "보류" 액션
- **보류함 페이지** (`/pending`): 두 그룹으로 표시
  - 보류된 ISSUE: 접을 수 있는 묶음 ("○○ 프로젝트 · task 7개"), 액션은 "통째로 복귀"만
  - 개별 보류 task: 카드 리스트 (sub-task 함께 표시), 액션 "인박스로 복귀"
  - 보류 시각 표시 ("3주 전 보류")
- 휴지통 페이지 패턴(`/tasks/trash`)을 재사용

### API

- `PATCH /api/tasks/[id]` 확장 또는 전용 라우트로 pending set/clear (하위 전파 포함, 서버에서 일괄 처리)
- `PATCH /api/issues/[id]` 동일
- 기존 GET 목록 쿼리들은 기본적으로 `pending_at is null` 필터 추가, 보류함 페이지만 `pending_at is not null`

## 2. 우선순위 필드 제거

- `tasks.priority` 컬럼 drop (마이그레이션 005에 포함). 기존 값은 폐기 — 사용자 확인 완료
- 코드 제거 범위:
  - `Priority` 타입, `Task.priority` (types.ts)
  - 인박스 필터에서 우선순위 항목 (5종 → 4종: 출처/상태/요청자/위임자)
  - 정렬에서 우선순위순 (6종 → 5종)
  - task 작성 폼·인라인 에디터·카드·상세 패널의 우선순위 UI/뱃지
  - Slack webhook·Notion sync의 priority 기본값 지정
  - 커스텀 뷰(custom-views) 저장값에 priority 참조가 있으면 마이그레이션 처리 (무시하고 버림)

## 3. 무게 인박스 (마감일 기반 시각 위계)

### 무게 판정 (클라이언트 계산, 저장 안 함)

| 티어 | 조건 | 표현 |
|---|---|---|
| **heavy** | 마감 지남 또는 오늘 | 제목 크고 굵게(700+), 왼쪽 3px 키컬러(#7D74F8) 라인 |
| **normal** | 마감이 오늘+7일 이내 | 현재 카드 스타일 유지 |
| **light** | 마감 없음 또는 7일 초과 | 작게·흐리게(muted) — 시야에서 물러남 |

* light의 점선 테두리는 v3 장식 최소 원칙에 따라 구현에서 제외 (2026-06-03)

- 완료/위임/취소 task는 기존 처리(line-through·opacity) 우선, 무게 스타일 미적용
- 디자인 시스템 v3 준수: 그림자 없음, 한 화면 액센트 1개(키컬러 라인이 그 1개), bg/border로만 위계
- 적용 범위: TaskCard 공용 — 인박스 트리·/today·커스텀 뷰 모두 동일 적용 (구현 시 결정 2026-06-03: 카드 컴포넌트가 공용이라 화면별 분기보다 일관 적용이 단순·일관적)

### ISSUE 헤더 보강 (B안 차용)

- 인박스 트리의 ISSUE 줄에 **진행률 텍스트(완료 n / 전체 m)** + **가는 progress bar** (키컬러, /today 상단 progress bar와 동일한 미세 스타일)
- "어느 프로젝트에 일이 몰렸는지" 덩어리 가시성 확보

## 만들지 않는 것 (YAGNI)

- 보류 리마인드·뱃지·자동복귀 ❌
- 보류된 ISSUE에서 개별 task 꺼내기 ❌
- 레이아웃 교체(칸반·타일 등) ❌ — 트리 구조 유지
- priority의 대체 수동 플래그(별표 등) ❌ — 마감일로 일원화

## 테스트 / 검증 포인트

- 보류 → 인박스·/today·카운트에서 제외되는지
- sub-task 동반 이동/복귀, ISSUE 통째 이동/복귀
- 휴지통과의 상호작용: 보류된 task 삭제 시 휴지통으로, 복원 시 보류 상태 유지 여부 (복원하면 보류도 해제하고 인박스로 — 단순하게)
- priority 제거 후 빌드·기존 저장 뷰·필터 동작
- 무게 티어 경계: 오늘 마감(heavy), 내일(normal), 8일 뒤(light), 마감 없음(light)
